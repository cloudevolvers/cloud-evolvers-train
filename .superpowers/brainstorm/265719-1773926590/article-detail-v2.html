Exchange Online RBAC and Access Policies: Locking Down Mailbox Access

Yair Knijn · March 20, 2026 · 9 min read
Exchange Online RBAC Security

Exchange Online has its own RBAC system, separate from Entra ID roles. Most admins discover this the hard way: they assign Global Admin to someone and that person still cannot run Get-Mailbox in PowerShell. This post covers the Exchange RBAC model, application access policies, and how to scope third-party app access to specific mailboxes.

Exchange RBAC: How It Actually Works

Exchange RBAC predates Entra ID roles. It uses role groups, management roles, and role assignments to control who can do what. The built-in role groups — Organization Management, Recipient Management, Help Desk — map to common admin personas.

Key point

Exchange RBAC is separate from Entra ID roles. Global Admin gives you Exchange admin access indirectly, but Exchange-specific roles give you granular control over mailbox operations.

The catch: you cannot use the Entra portal to assign Exchange roles. You need the Exchange Admin Center or PowerShell. And the role names do not map 1:1 to what you see in Entra.

▼ List Exchange role group members PowerShell
📋 Copy 
# Connect to Exchange Online
Connect-ExchangeOnline -UserPrincipalName admin@contoso.com

# List all role group members
Get-RoleGroupMember -Identity "Organization Management"

# Check a specific user's roles
Get-ManagementRoleAssignment -RoleAssignee "user@contoso.com"

Application Access Policies

When a third-party app registers in Entra ID and gets the full_access_as_app permission for Exchange, it can read every mailbox in the tenant. Application access policies let you scope that down to specific mailboxes.

Warning

Without an access policy, any app with EWS or Graph mailbox permissions can read all mailboxes. This is the default. You have to explicitly restrict it.
